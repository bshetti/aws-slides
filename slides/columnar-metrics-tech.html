Blazing Fast Query
Performance
Delivers faster AI investigations
  • TS: inner agg per time series, then aggregated by group using a secondary aggregation function — expresses TS math in one ES|QL pipeline.
  • Sorted _tsid + time, vectorized + parallel ES|QL — sequential column reads and higher CPU utilization on hot queries.
  • Codec → engine primitive blocks (zero-copy) + constant blocks — fewer copies vs row materialization; less allocator churn.
  • Push down time / dim filters to Lucene skippers — skips non-matching blocks before decode; lower IO and CPU on narrowed ranges.
  • RATE() on ordered _tsid slices (partitioned by their prefix) + bucket interpolation — correct counter resets and boundary deltas with fewer passes.
  • Sliding windows: bucket partials, then span merge partial results — smoother trend lines without rereading all points each window step.
Storage Efficiency Gain
Faster root cause. Store more for richer AI context at lower cost
  • Doc-value skippers replace inverted indices + BKD trees on @timestamp + dims — true columnar files; removes duplicate per field index bulk overhead.
  • Synthetic _id = _tsid+@timestamp with skipper + bloom checks — drops a heavy _id inverted index metrics rarely need.
  • Trimmed sequence numbers + relaxed OCC on TSDS where concurrent doc updates are uncommon — smaller postings and replication metadata.
  • Numeric codec with 512 byte blocks (was 128) for dimensions with IP/MAC addresses — tighter compression when long runs of those bytes repeat (the 128-wide blocks missed the pattern).
  • Shard route + index sort on _tsid — keeps each series contiguous on disk so codecs and skippers stay sequential.
Prometheus Ingest & PromQL
Zero friction to switch. Engineers using Grafana feel right at home
  • Native OTLP + Prometheus remote-write protobuf — cheaper parse and larger batches than JSON _bulk translation paths.
  • Coordinator evaluates _tsid once per indexed document and forwards it — data nodes skip repeating the same dimension-hash / series-id work for every doc.
  • Hash dimensions once per protobuf batch — amortizes _tsid work across many points in a single network frame.
  • Skipper-backed metric segments — lighter merges vs maintaining hot inverted structures during sustained ingest.
  • Synthetic recovery source — replicate metrics without persisting stored _source; materially lowers recovery-source disk IO.
  • Same TSDS column files for PromQL + ES|QL — one storage layout powers both surfaces.
